# SERA Engine vNext ADR v0.2.0
Status: ACCEPTED_FOR_PLANNING  
Phase: A4+R-26 — Legacy Engine Freeze, Backup and SERA Engine vNext Clean Start  
Scope: architectural decision to freeze the legacy SERA engine and design a clean vNext causal engine  
Non-scope: code implementation, fixture creation, baseline promotion, risk-layer redesign, HFACS integration, A5 Risk Layer
---
## 1. Purpose
This ADR records the decision to stop patching the current legacy SERA engine as the methodological source of truth and to create a clean SERA Engine vNext.
The goal is to avoid carrying forward architectural contamination from the old engine.
The legacy engine will be preserved for traceability, comparison and containment, but vNext will be designed from first principles around the SERA causal workflow.
---
## 2. Decision summary
Decision:
```text
Freeze current SERA engine as LEGACY_CONTAINMENT.
Create SERA Engine vNext as isolated causal engine.
Keep Risk Layer, HFACS and dashboards outside the causal engine.
Use real-event trials as acceptance tests.
Do not continue remediating legacy Step6/7 as the long-term path.
```

---
## 3. Why this decision was made

The first real-event trial, TRIAL-SET1-001 / REAL-EVENT-0001, demonstrated that the legacy engine still carries methodological contamination.

Observed failures:

* P-A was selected as "no perception failure", but conclusion described a perception/detection error.
* O-A was selected as "no objective failure", but conclusion described a routine violation / inadequate objective.
* A-D was selected without evidence of physical, motor or ergonomic inability.
* Preconditions in the conclusion diverged from the deterministic preconditions list.
* HFACS was rendered from an unsupported A-D code.
* Risk/ERC was displayed as if the causal analysis were valid.
* Escape point was framed too strongly as a decision instead of separating decision antecedent from operational unsafe state.

A patch was applied in commit:

7398bc0 fix(sera): add guardrails for trial 001 causal consistency

However, the rerun still showed the same real-path contamination.

Therefore, further patching of the legacy engine risks accumulating technical debt without guaranteeing methodological correctness.

## 4. Legacy engine status

The current engine is reclassified as:

LEGACY_SERA_ENGINE_v0.1.4_CONTAINMENT

Allowed uses:

* historical traceability;
* regression comparison;
* UI continuity during transition;
* reference for known failure modes;

Forbidden uses:

* source of truth for methodology;
* basis for real-event validation;
* basis for risk-layer design;

* use Git references/tags for traceability;
* use archive output only if needed for external/offline backup.

## 6. Clean start principles for vNext

SERA Engine vNext must start from a clean methodological contract.

vNext must not inherit:

* free LLM final conclusion before normalized causal state;
* Step6/7 unconstrained "falhas ativas" narrative;
* automatic HFACS downstream mapping;
* automatic Risk/ERC downstream mapping;
* preconditions generated by one path and displayed by another without reconciliation;
* decision-only escape point framing.

vNext may reuse:

* TypeScript project infrastructure;
* existing test runner patterns;
* baseline/candidate discipline;
* real-event harvest documents;
* source governance documents;
* UI integration later through adapter;
* lessons learned from legacy failure reports.

---
## 7. vNext causal workflow

The vNext engine must implement this sequence explicitly:

1. neutral factual input
2. factual extraction
3. operational unsafe state / safe operation escape point
4. decision antecedents, if any
5. unsafe act and unsafe condition separation
6. direct actor identification
7. perception statement
8. objective/goal statement
9. action statement
10. P/O/A classification
11. preconditions
12. limitations and unanswered questions
13. recommendations
14. human review status

No final conclusion may be generated before steps 1–13 are normalized.

---
## 8. Required separation of concerns

The causal engine must output only causal analysis.

Out of scope for the causal engine:

HFACS mapping
ARMS/ERC
traditional risk matrix
risk profile aggregation
dashboards
corrective-action management
commercial report formatting

These may exist as separate layers after causal validation.

## 10. Acceptance tests for vNext

Initial acceptance set:

TRIAL-SET1-001 / REAL-EVENT-0001

Must pass before trials 002–005 are reopened.

Minimum pass criteria for Trial 001:

* no unsupported A-D;
* no P-A described as perception failure;
* no O-A described as objective failure;
* no A-A described as action failure;
* no HFACS output from causal engine;
* no Risk/ERC output from causal engine;
* limitations preserved;
* unsafe condition and warning/barrier issue preserved;
* operational unsafe state separated from decision antecedent.

---
## 11. Migration strategy

Use a parallel/strangler strategy:

legacy engine remains available but marked legacy
vNext engine built in isolated module
adapter compares legacy vs vNext
UI can later switch by feature flag
risk/HFACS reintroduced only after vNext causal assurance
legacy retired after acceptance tests and human review

Do not replace the entire application at once.

Do not delete legacy code in the first vNext phase.

## 15. Stop rules

Do not proceed to vNext implementation if:

* legacy state has not been preserved;
* vNext scope still includes risk/HFACS;
* output contract is not agreed;
* Trial 001 acceptance criteria are unclear;
* implementation would modify legacy engine directly;
