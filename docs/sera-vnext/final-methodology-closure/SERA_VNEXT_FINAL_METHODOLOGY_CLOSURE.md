## Practical Meaning

1. Do not create new methodology-preparation phases, readiness packets, or intermediate closure layers.
2. Treat future work on UI, API, auth, persistence, audit, monitoring, deploy, and pilot operations as product work.
3. Keep scientific external validation as optional backlog, not as a product blocker.

## Allowed Reasons To Reopen Methodology

Methodology may change only when at least one of the following is true:

1. a real methodological error is identified;
2. a new authorial decision explicitly changes the causal baseline;
3. a taxonomic change must be absorbed;
4. materially relevant evidence requires revision;
5. a validated regression proves the current baseline is broken.

## Explicit Non-Reopeners

The following do not reopen methodology by themselves:

1. product UI work;
2. endpoint work;
3. auth integration work;
4. candidate-only runtime work;
5. persistence backlog planning;
6. optional academic validation backlog.
